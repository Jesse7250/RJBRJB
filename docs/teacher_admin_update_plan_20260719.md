# 智学蜂巢教师端与管理端更新计划文档（2026-07-19）

## 1. 设计目标

当前项目已经形成学生端主流程：课程广场 -> 进入课程 -> 学习画像 / 知识图谱 / 学习资源 / 学习对话 / 代码沙箱 / 掌握进度，并保留“小蜂导学”数字人作为全局辅助入口。

本次新增教师端与管理端的目标，不是简单加两个后台表格页，而是把智学蜂巢扩展成三类用户共同使用的教学平台：

- 学生端：围绕“学什么、怎么学、哪里不会、如何练习”。
- 教师端：围绕“教什么、谁在学、哪里卡住、如何补救”。
- 管理端：围绕“有哪些课、有哪些人、权限是否正确、平台是否健康”。

设计原则：保留学生端已经建立起来的浅色教育平台气质、蜂巢知识网络视觉语言和数字人能力，但教师端与管理端要更偏工作台、数据看板和流程管理，避免过度装饰。

## 2. 参考页面理解

参考智慧树登录与教师使用入口后，登录设计可以借鉴以下方向：

- 登录中心提供机构登录、账号登录、学号登录、工号登录、扫码登录等多种身份入口。
- 教师身份常见登录方式包括手机号登录、教师工号登录；教师工号登录通常依赖学校/机构数据联动。
- 教师端不仅是“看课”，还包括课程建设、课程资料、题库、作业考试、学生管理、学习进度、教学运行等工作流。

## 3. 登录与角色分流设计

### 3.1 登录入口总体结构

新增统一登录页，不再让课程广场里的登录框承担所有身份逻辑。

建议路由：

```text
/#/login/student
/#/login/teacher
/#/login/admin
/#/portal
/#/course/python/graph
/#/teacher
/#/admin
```

登录页首屏结构：

- 左侧：智学蜂巢品牌区，展示平台定位、课程智能体、知识图谱、数字人导学。
- 右侧：身份登录面板，使用分段标签切换“学生 / 教师 / 管理员”。
- 底部：帮助入口，例如“忘记密码”“联系管理员”“教师入驻申请”。
- 数字人：保留在登录页右下或面板旁边，负责解释不同身份如何登录。

### 3.2 学生登录方式

学生端保留现有注册 / 登录逻辑，同时扩展为更像真实平台的入口：

- 手机号 / 用户名登录。
- 学号登录：学校、学号、密码。
- 注册入口：面向自由注册学生。

登录成功后：

- 跳转课程广场。
- 学生选择课程后进入学生学习工作台。

### 3.3 教师登录方式

教师端参考智慧树教师登录逻辑，建议提供两种主要方式：

- 手机号 / 用户名登录。
- 教师工号登录：学校、教师工号、密码。

教师首次使用流程：

1. 点击“教师登录”。
2. 选择“手机号登录”或“教师工号登录”。
3. 如果教师账号未认证，进入“教师认证 / 绑定学校”流程。
4. 登录成功后进入教师工作台。

教师注册不建议像学生一样完全开放。建议改为：

- “教师入驻申请”。
- 填写学校、工号、姓名、手机号、邮箱。
- 管理员审核通过后开通教师权限。

### 3.4 管理员登录方式

管理端不建议开放注册。

管理员登录方式：

- 管理员账号 + 密码。
- 可选二次验证：验证码 / 邮箱验证码 / 管理员口令。

入口展示建议：

- 不在课程广场主视觉里强展示。
- 登录页角色切换中保留“管理员”入口。
- 管理员登录失败提示要克制，不暴露账号是否存在。

### 3.5 登录后权限逻辑

后端 token 中需要带角色信息，前端根据角色决定跳转：

```text
student -> /#/portal
teacher -> /#/teacher
admin   -> /#/admin
```

前端路由守卫：

- 未登录访问教师端 / 管理端：跳转对应登录页。
- 学生访问教师端：显示无权限页。
- 教师访问管理端：显示无权限页。
- 管理员可进入管理端，但不默认进入教师授课工作台。

## 4. 教师端页面设计

### 4.1 教师端整体信息架构

教师端建议使用与学生课程工作台相近的顶部课程信息区，但左侧导航换成教学管理功能。

教师端导航：

```text
教师首页
我的课程
课程建设
学生学情
习题批阅
资源管理
班级热力图
教学通知
数字人助手
```

核心页面不建议全部塞在一页。和学生端一样，每个功能模块独立页面，通过左侧导航或顶部 tab 切换。

### 4.2 教师首页

用途：让教师一进来就知道今天该处理什么。

布局：

- 顶部欢迎区：教师姓名、所属学校、当前授课课程数。
- 左侧主区域：我的课程卡片列表。
- 右侧辅助区：小蜂导学教师版、待处理事项、系统提示。
- 下方数据区：近期学习活跃、待批阅习题、低掌握知识点、资源审核状态。

数据必须来自后端。若后端没有数据，不显示假统计，改为空状态：

```text
暂无待处理任务
暂无学生学习记录
暂无已发布课程
```

### 4.3 我的课程

教师可以查看自己负责的课程。

课程卡片字段：

- 课程名称。
- 所属分类。
- 授课班级。
- 学生人数。
- 课程状态：草稿 / 已发布 / 运行中 / 已归档。
- 最近更新时间。
- 快捷操作：进入课程、编辑课程、上传资源、查看学情。

### 4.4 添加课程 / 课程建设

教师端必须支持添加课程，这是新增端的核心功能之一。

建议做成分步式建课向导：

1. 基本信息
   - 课程名称。
   - 课程分类。
   - 课程简介。
   - 适用对象。
   - 课程封面。

2. 章节与知识点
   - 添加章节。
   - 添加知识点。
   - 设置知识点难度。
   - 设置前置知识点关系。
   - 可选择“AI 辅助生成知识图谱草稿”。

3. 学习资源
   - 上传 PDF / Word / Markdown / PPT / 视频 / 音频。
   - 将资源绑定到章节或知识点。
   - 设置资源类型：讲义、案例、练习题、视频、音频、参考资料。

4. 练习与评价
   - 添加选择题、填空题、简答题、代码题。
   - 设置参考答案、评分规则、知识点归属。
   - 对代码题可配置输入样例、期望输出、判题规则。

5. 发布确认
   - 展示课程完整度检查。
   - 检查是否存在未绑定知识点、无资源章节、无题目章节。
   - 发布课程或保存草稿。

视觉设计：

- 使用浅色分步条，当前步骤用蜂巢绿色高亮。
- 右侧固定“小蜂检查清单”，提示哪些内容还未完成。
- 每一步都支持保存草稿，避免教师填写中断。

### 4.5 学生学情

教师可以查看该教师课程下学生的学习情况。

筛选条件：

- 课程。
- 班级。
- 学生。
- 时间范围。
- 知识点。
- 掌握度区间。

主视图：

- 班级概览：平均掌握度、活跃学生数、风险学生数、待批阅数。
- 学生列表：姓名、学号、学习时长、完成资源数、练习正确率、最近学习时间、当前卡点。
- 点击学生行后打开学生详情侧栏。

学生详情侧栏：

- 学习画像摘要。
- 当前学习路径。
- 已掌握 / 未掌握知识点。
- 最近练习记录。
- AI 对话摘要。
- 教师备注。

### 4.6 习题及作答查看

教师端应能查看学生习题和代码练习情况。

页面结构：

- 左侧筛选：课程、章节、知识点、题型、是否已批阅。
- 中间列表：学生提交记录。
- 右侧详情：题目、学生答案、参考答案、判题结果、变量快照、运行输出、错误信息。

代码题详情建议复用学生端代码沙箱的视觉语言，但教师端默认只读：

- 代码区展示学生提交代码。
- 输出区展示运行结果。
- 变量快照展示后端返回变量。
- 教师可写评语和手动调整评分。

### 4.7 班级热力图

学生端的热力图适合个人自查，教师端热力图要变成班级诊断工具。

建议提供两种热力图：

1. 学生 x 知识点热力图
   - 行：学生。
   - 列：知识点。
   - 颜色：掌握度。
   - 点击单元格：查看该学生在该知识点的练习、错误、资源访问记录。

2. 时间 x 知识点热力图
   - 行：日期 / 周次。
   - 列：知识点。
   - 颜色：班级整体掌握变化。
   - 用于判断某个知识点是否需要课堂补讲。

颜色说明要用户能看懂：

```text
绿色：掌握较好
黄色：需要巩固
红色：风险较高
灰色：暂无学习记录
```

### 4.8 资源管理 / 上传课程资源

教师可以上传课程相关学习资源，并与知识点绑定。

功能：

- 上传文件：PDF、DOCX、PPTX、Markdown、图片、视频、音频。
- 填写资源标题、简介、适用章节、适用知识点。
- 选择是否允许 AI 根据该资源生成讲义、导图、练习题、听觉型讲解稿。
- 查看资源解析状态：待解析 / 解析中 / 已解析 / 解析失败。
- 查看资源审核状态：待审核 / 通过 / 需修改。

资源上传后流程：

```text
教师上传资源 -> 后端存储文件 -> 文档解析 -> 知识点绑定 -> AI 生成候选资源 -> Reviewer 审核 -> 教师确认发布 -> 学生端可见
```

### 4.9 教师端数字人

教师端保留数字人，但定位与学生端不同。

教师端数字人建议叫：

```text
小蜂助教 · 教学助手
```

能力侧重：

- 解释教师端页面怎么用。
- 帮教师解读班级热力图。
- 根据学生薄弱点建议补充讲解。
- 根据课程资料建议可生成哪些学习资源。
- 帮教师草拟课程通知。
- 引导教师完成课程建设清单。

不建议让教师端数字人直接编造学生数据。如果后端没有数据，应回答：

```text
当前课程还没有学生学习记录，我可以先帮你检查课程资源是否完整。
```

## 5. 管理端页面设计

### 5.1 管理端整体信息架构

管理端面向平台管理员，视觉应更克制，偏运营和权限管理。

管理端导航：

```text
平台总览
课程管理
教师管理
学生管理
班级与选课
资源审核
系统监控
权限与日志
数字人助手
```

### 5.2 平台总览

用途：快速判断系统是否健康、平台是否正常运行。

模块：

- 总课程数。
- 教师数。
- 学生数。
- 今日活跃会话数。
- 资源生成任务数。
- TTS 状态。
- LLM 状态。
- 图谱后端状态。

现有后端已有 `/api/admin/stats`，但数据维度还不够，需要扩展为角色、课程、系统服务状态更完整的统计。

### 5.3 课程管理

管理员可以管理所有课程。

功能：

- 查看所有课程。
- 新增课程。
- 编辑课程基础信息。
- 设置课程状态：草稿 / 已发布 / 暂停 / 归档。
- 分配课程负责人和教学团队。
- 查看课程资源数量、学生数量、运行状态。
- 对违规或质量较低课程进行下架。

### 5.4 教师管理

功能：

- 查看教师列表。
- 新增教师账号。
- 审核教师入驻申请。
- 绑定学校 / 院系 / 工号。
- 分配教师到课程。
- 重置密码。
- 禁用 / 启用账号。

### 5.5 学生管理

功能：

- 查看学生列表。
- 按课程、班级、学校筛选。
- 查看学生学习概览。
- 重置学生密码。
- 调整学生所在班级。
- 查看异常学习记录。

### 5.6 班级与选课管理

功能：

- 创建班级。
- 将学生加入班级。
- 将班级绑定课程。
- 批量导入学生名单。
- 导出选课名单。
- 查看班级学习概况。

### 5.7 资源审核

管理员可以查看平台级资源生成与审核状态。

功能：

- 查看所有教师上传资源。
- 查看 AI 生成资源版本。
- 查看 Reviewer / Guardian 审核结果。
- 手动触发资源重审。
- 下架不合格资源。

现有 `/api/admin/resource-review` 可以作为基础，但需要扩展资源列表、审核记录、操作日志。

### 5.8 系统监控

管理端需要帮助判断为什么“后端中断”“TTS 不可用”“模型超时”。

建议展示：

- 后端健康状态。
- DeepSeek 配置状态。
- 讯飞 TTS 配置状态。
- 图谱后端：memory / neo4j。
- 数据库连接状态。
- 最近 Agent 任务失败原因。
- 资源生成平均耗时。

### 5.9 管理端数字人

管理端也保留数字人，但定位为平台运营助手。

能力侧重：

- 解释各管理模块用途。
- 帮管理员排查接口异常。
- 汇总平台运行状态。
- 提醒哪些课程资源审核失败。
- 提醒哪些教师账号待认证。

## 7. 本期真实实现范围

本期教师端和管理端按“轻量但真实可用”实现，不做大而全的后台，不展示当前后端无法支撑的数据。

### 7.1 本期保留的真实功能

```text
学生：继续使用现有学生端课程广场与 Python 学习工作台。
教师：登录后进入教师端，能够新增课程、查看自己创建的课程、编辑课程基础信息。
管理员：登录后进入管理端，能够查看真实用户列表、真实课程列表，并管理课程状态。
数字人：学生端、教师端、管理端都保留小蜂导学，但按当前角色解释页面功能。
```

本期教师端不展示假的学生列表、假习题记录、假班级热力图。只有当数据库中存在真实学生学习事件、真实课程归属和真实提交记录后，才扩展对应页面。

### 7.2 本期登录分流

登录入口保留三类身份：

```text
学生登录：进入课程广场。
教师登录：进入教师工作台。
管理员登录：进入管理工作台。
```

后端用户表增加 `role` 字段，取值为：

```text
student
teacher
admin
```

前端根据登录返回的 `role` 做跳转，不靠前端写死身份。

### 7.3 本期课程落库

教师新增课程必须写入数据库。课程刷新后仍然存在。

课程最小字段：

```text
course_id
Title
category
summary
teacher_username
status
created_at
updated_at
```

教师端只展示当前教师创建的课程。管理员端展示所有课程。

## 8. 本期后端接口范围

只新增当前页面真正使用的接口。

### 8.1 认证接口

```text
POST /api/auth/register
POST /api/auth/login
GET  /api/auth/me
```

说明：

- 注册接口支持 `role` 字段。
- 登录返回 `username` 和 `role`。
- `me` 接口用于刷新页面后恢复当前用户身份。

### 8.2 教师课程接口

```text
GET   /api/teacher/courses
POST  /api/teacher/courses
PATCH /api/teacher/courses/{course_id}
```

说明：

- 只能读取和修改当前教师自己的课程。
- 新增课程会写入 SQLite 数据库。
- 不做假的学生统计、假的习题数、假的热力图。

### 8.3 管理端接口

说明：

## 10. 数字人角色化方案

数字人不应在三个端里回答完全一样的话。建议后端 `SYSTEM_PROMPT` 根据 `roleContext` 注入不同角色定位。

### 学生端

定位：学习陪伴与页面导学。

可以回答：

- 这个页面怎么用。
- 我现在应该学哪个知识点。
- 学习资源、知识图谱、代码沙箱怎么配合使用。

### 教师端

定位：教学管理助手。

可以回答：

- 如何添加课程。
- 如何查看学生薄弱知识点。
- 如何上传并审核课程资源。
- 如何根据热力图安排补讲。

### 管理端

定位：平台运营助手。

可以回答：

- 如何管理课程和用户。
- 如何排查 TTS / LLM / 图谱服务状态。
- 如何查看资源审核失败原因。
- 如何处理教师入驻申请。

## 10. 视觉设计方向

### 10.1 整体主题

继续沿用当前学生端浅色主题：

- 主色：蜂巢绿 / 智慧树风格绿色。
- 辅色：暖金色，用于关键路径、强调状态、资源生成状态。
- 背景：浅绿白、轻纹理、柔和边界。
- 卡片：白底、轻阴影、低圆角，不回到深色主题。

### 10.2 教师端视觉

教师端应更像“教学驾驶舱”：

- 数据卡片密度比学生端高。
- 图表区更大。
- 操作按钮明确，例如“添加课程”“上传资源”“查看学情”。
- 热力图作为视觉重点，放在页面主区域。

### 10.3 管理端视觉

管理端应更像“平台控制台”：

- 左侧导航更稳定，不做太多花哨动效。
- 表格、筛选器、状态标签要清晰。
- 系统监控页可以保留轻微动态，例如服务状态呼吸点、任务流转时间线。

### 10.4 登录页视觉

登录页参考智慧树教师登录，但要保持智学蜂巢主题。

建议布局：

```text
左侧：品牌与数字人说明区
右侧：身份登录卡片
底部：帮助与协议
```

交互：

- 切换学生 / 教师 / 管理员时，登录字段同步变化。
- 教师工号登录时出现“学校 + 工号 + 密码”。
- 管理员登录时出现更严肃的安全提示。
- 数字人根据当前登录身份给出不同提示。

## 11. 本期实际实施范围

本期只做能真实落库、真实读取、刷新后仍存在的功能。教师端和管理端是扩展入口，不作为主线功能堆复杂后台。

### 11.1 已按真实接口实施

- 登录 / 注册增加 `student`、`teacher`、`admin` 三类角色，后端 token 返回角色信息。
- 前端根据登录角色进入学生课程广场、教师工作台或管理工作台。
- 教师端提供课程创建、课程列表、课程状态更新，数据写入后端 SQLite `courses` 表。
- 管理端读取真实用户列表、真实课程列表和数据库统计，并支持课程状态更新。
- 用户表补充 `role` 字段，旧数据库启动时自动补列，避免角色信息丢失。

### 11.2 不展示的内容

以下内容没有完整真实数据链路，本期不展示，也不做假数据占位：

- 假学生名单。
- 假班级热力图。
- 假习题提交记录。
- 假资源审核记录。
- 假平台运营统计。
- 未接入真实提交记录的教师批阅入口。
- 未接入真实班级关系的班级热力图入口。
- 未接入真实资源上传解析链路的课程资源上传入口。

### 11.3 验收标准

- 教师账号登录后进入教师端。
- 管理员账号登录后进入管理端。
- 教师新增课程后，数据库里能查到。
- 教师刷新页面后，课程仍然存在。
- 管理员能看到真实用户和真实课程。
- 没有真实数据时显示空状态，不用演示数据凑页面。
